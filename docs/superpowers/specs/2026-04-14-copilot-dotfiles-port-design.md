# Copilot Dotfiles Port Design

Port the claude-dotfiles system to GitHub Copilot CLI with one-way sync from claude-dotfiles (canonical source) to copilot-dotfiles (generated output).

## Context

The user maintains a comprehensive Claude Code configuration system (claude-dotfiles) with 35+ slash commands, an MCP session-context server, a harness system for multi-session work, a tape-based continuity system, project registry, and templates. They need to use Copilot CLI at work and want to bring forward as much of this system as possible.

Copilot CLI (GA since Feb 2026, `@github/copilot` v1.0.26+) has close analogs for nearly every Claude Code concept: skills (SKILL.md), agents (.agent.md), hooks, MCP servers, config.json, and a plugin ecosystem.

## Architecture

### Source of Truth

claude-dotfiles is canonical. A sync command (`just sync-copilot`) runs a Node.js transform that generates the full copilot-dotfiles tree. copilot-dotfiles is never hand-edited — a `.copilot-generated` marker file signals this.

### Sync Direction

One-way: claude-dotfiles → copilot-dotfiles. No back-sync.

### Plugin Management

A shared `plugins.yaml` in claude-dotfiles drives plugin installation for both platforms. Each install script reads this manifest and ensures required plugins are present.

### Design Decisions

- Commands become **skills** (SKILL.md), not agents. Skills are the Copilot analog of Claude commands — invoked by name, contain instructions.
- Short aliases (ws, wd, etc.) become lightweight **agents** (.agent.md) that delegate to the skill.
- The MCP session-context server is shared as-is — it's tool-agnostic.
- `AGENTS.md` is the primary instructions file (cross-tool standard; Copilot also reads CLAUDE.md but AGENTS.md is canonical for Copilot).

## Install Script

Doctor-mode installer, same pattern as claude-dotfiles.

### Prerequisites

| Prereq | Check |
|---|---|
| Node.js | `node --version` |
| Git | `git --version` |
| `copilot` CLI | `copilot --version` |
| `just` | `just --version` |
| `gh` CLI | `gh --version` |

### Phases

| Phase | What |
|---|---|
| 1. Prerequisites | Validate all prereqs present |
| 2. Symlinks | `copilot-instructions.md` → `~/.copilot/`, `skills/` → `~/.copilot/skills/`, `agents/` → `~/.copilot/agents/` |
| 3. Config merge | Merge `config.json` into `~/.copilot/config.json` (don't clobber user additions) |
| 4. MCP registration | Register session-context in `~/.copilot/mcp-config.json` |
| 5. Plugins | Install required plugins from manifest (superpowers, etc.) |
| 6. Machine identity | Verify `~/.machine-id` exists (shared with claude-dotfiles) |
| 7. MCP server build | `cd mcp/session-context && npm install && npm run build` |

### Flags

- `--dry-run` — show what would change
- `--force` — auto-fix without prompting
- `--uninstall` — remove symlinks only

## Plugin Manifest

Shared manifest in claude-dotfiles (`plugins.yaml`) drives both platforms:

```yaml
plugins:
  superpowers:
    claude:
      source: obra/superpowers
    copilot:
      marketplace: superpowers-marketplace
      marketplace_source: obra/superpowers-marketplace

  playwright:
    type: cli
    package: "@playwright/cli@latest"
    install: "npm install -g @playwright/cli"

mcp_servers:
  session-context:
    command: node
    args: ["mcp/session-context/dist/index.js"]
    relative: true
```

Both install scripts read this manifest. Claude's handles the `claude` key, Copilot's handles the `copilot` key, shared entries (playwright CLI, MCP servers) are handled by both.

### Location

`sync/copilot-sync.mjs` in claude-dotfiles, invoked via `just sync-copilot`.

Node.js script. Node is already a prerequisite (MCP server), handles YAML parsing, template generation, and file I/O. No new dependencies beyond what the MCP server already requires.

### Transform Rules

#### Command → SKILL.md

1. Extract first paragraph after `# Title` as the `description` frontmatter value
2. Pull `allowed-tools` from the mapping config
3. Replace `## Instructions for Claude` → `## Instructions`
4. Apply `tool_map` substitutions in body text
5. Replace `Agent` tool references with `task`
6. Replace `~/.claude/` path references with `~/.copilot/`
7. Leave MCP tool names as-is (`mcp__session-context__*` — same server)
8. Leave everything else untouched

#### Alias → .agent.md

For each command with an alias, generate:

```markdown
---
name: <alias>
description: Alias for <skill-name> — <description>
---

Run the <skill-name> skill. Pass through any arguments the user provided.
```

#### CLAUDE.md → AGENTS.md

Full content with:
- "Instructions for Claude" → "Instructions"
- `.claude/` paths → `.copilot/` paths
- Tool name substitutions per tool_map

#### CLAUDE.md → copilot-instructions.md

Extract user-level sections only:
- Who I Am
- How I Work (Development Philosophy, Communication Style, When You're Stuck)
- Universal Preferences (Code Style, Git Practices, Testing)
- Stack-Specific Notes
- What I Don't Want

Omit harness, orchestration, registry, tape sections.

### Sync Flow

```
1. Load copilot-mapping.yaml
2. Load plugins.yaml
3. Clean output dir (copilot-dotfiles/) — preserve .git
4. For each command in mapping:
   a. Read commands/<name>.md
   b. Apply tool_map substitutions
   c. Generate SKILL.md frontmatter
   d. Write to skills/<name>/SKILL.md
   e. If alias defined, write agents/<alias>.agent.md
5. Transform CLAUDE.md → AGENTS.md
6. Transform CLAUDE.md → copilot-instructions.md (user-level extract)
7. Generate config.json (settings + hooks)
8. Generate mcp-config.json (from plugins.yaml mcp_servers)
9. Copy shared assets (mcp/, templates/, registry/, lib/)
10. Generate install.sh (from template, with plugin manifest)
11. Write .copilot-generated marker
12. Report: "Synced N skills, N agents, N shared assets"
```

Idempotent — running twice produces identical output.

## Config Transform

### config.json

```json
{
  "model": "claude-sonnet-4-6",
  "statusLine": "node ~/.copilot-dotfiles/lib/statusline.cjs",
  "includeCoAuthoredBy": true,
  "hooks": {
    "sessionStart": [],
    "preToolUse": [],
    "postToolUse": []
  }
}
```

### Hooks Migration

Claude format:
```json
{ "hooks": { "PreToolUse": [{ "matcher": "Bash", "command": "./guard.sh" }] } }
```

Copilot format:
```json
{ "hooks": { "preToolUse": [{ "type": "command", "bash": "./guard.sh" }] } }
```

### Statusline

`lib/statusline.cjs` reads session JSON via stdin. During implementation, verify the Copilot CLI statusline input contract matches Claude Code's. If different, add format detection to handle both, or fork to `statusline-copilot.cjs`. Same visual output either way. Must-have — not optional.

## Pre-Sync Changes to claude-dotfiles

Before the first sync, these changes are needed in claude-dotfiles:

1. **Update `work-exec.md`** — Replace fire-and-forget wave dispatch with delegation to `superpowers:executing-plans`. Simplifies the command and makes it portable across both platforms.

2. **Add sync infrastructure:**
   - `sync/copilot-sync.mjs` — the transform script
   - `sync/copilot-mapping.yaml` — command mapping + tool translations
   - `plugins.yaml` — plugin manifest for both platforms

3. **Add `just sync-copilot` target** to claude-dotfiles justfile

4. **Update `install.sh`** — Add `just` as a prerequisite. Add plugin management (read plugins.yaml, ensure Claude Code plugins installed, ensure playwright CLI installed).

These are additions — nothing breaks for current Claude Code usage.

## Command Scope

### Ported (20 commands + 10 aliases)

**Session lifecycle:** work-init, work-start (ws), work-done (wd), work-status (wst)

**Planning & execution:** work-plan (wp), work-exec (we), work-deep (wdp)

**Issue management:** work-create (wc), issue-close

**Parallel dev:** work-branch, work-merge

**Maintenance:** housekeeping (hk), work-archive (wa), tape-migrate

**Project scaffolding:** project-init (pi)

**Wiki:** wiki-ingest, wiki-search, wiki-update, wiki-lint

**Testing:** visual-test

### Skipped

- overnight (deeply tied to autonomous loop model)
- backbone-onboard (infrastructure-specific)
- code-review (Copilot has built-in `/review`)

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Copilot statusline contract differs from Claude | Format detection in statusline.cjs, or fork |
| Superpowers plugin behaves differently on Copilot | Superpowers v5.0.7+ has native Copilot support with tool-name translation |
| MCP session-context server has platform-specific behavior | Server is tool-agnostic (stdio transport, no Claude-specific APIs) |
| Copilot skill invocation syntax differs from Claude commands | Skills are invoked by name in Copilot, same as Claude — `/work-start` works |
| Some command prose assumes Claude-specific capabilities | Tool map substitutions catch most; manual review for edge cases |
